## 2. Eliminate `features/ → audit/` dependency (cross-feature coupling)

**Status:** Implemented — see [ADR-003](docs/adr/003-eliminate-audit-coupling.md)

- **Files**: `features/events/resolvers/event-config-builder.ts` (imports `audit/plugin-integration`), `features/core/toast-director.ts` (imports `audit/plugin-integration`)
- **Problem**: Two feature modules (events, core) pull global audit state. Config builder logs errors via audit; toast director logs "toast dropped" via audit. Creates invisible coupling — swapping audit requires changing events and core.
- **Solution**: Inject required callbacks (`onUnknownEvent`, `onToastDropped`) as constructor/context parameters with no-op defaults. Entry point wires real audit calls after `initAuditLogging()`. No `?.` at call sites — typed as required.
- **Benefits**: **Leverage** — pure interfaces, no global module dependency. **Testability** — tests don't need to mock audit. **Real seam** — could swap audit system without touching events/core.

## 3. Unify config resolvers (`ConfigBuilder` + `ToolConfigResolverImpl`)

**Status:** Implemented — see [ADR-004](docs/adr/004-unify-config-resolvers.md)

- **Files**: `features/events/resolvers/event-config-builder.ts` (75 lines), `features/events/resolvers/tool-config.resolver.ts` (280 lines)
- **Problem**: `EventConfigResolverImpl` delegates to `ConfigBuilder`, but `ToolConfigResolverImpl` duplicates the same logic inline. Understanding config resolution requires jumping across 10+ files. Both classes use the `Impl` suffix anti-pattern which communicates nothing about their role.
- **Solution**: Extract `DefaultConfigResolver` class used by both via composition. Rename `EventConfigResolverImpl` → `DelegatingEventConfigResolver` and `ToolConfigResolverImpl` → `DefaultToolConfigResolver`.
- **Benefits**: **Locality** — single resolution logic in `DefaultConfigResolver`. **Leverage** — public interface stays the same. ~80 lines of duplication eliminated. Names communicate actual responsibility.
