## 2. Unify config resolvers (`ConfigBuilder` + `ToolConfigResolverImpl`)

**Status:** Open

- **Files**: `features/events/resolvers/event-config-builder.ts` (197 lines), `features/events/resolvers/tool-config.resolver.ts` (336 lines)
- **Problem**: `EventConfigResolverImpl` delegates to `ConfigBuilder`, but `ToolConfigResolverImpl` duplicates the same logic inline. Understanding config resolution requires jumping across 10+ files.
- **Solution**: Extract shared resolution logic into a `DefaultConfigResolver` used by both. `ToolConfigResolverImpl` becomes a thin adapter.
- **Benefits**: **Locality** — single resolution logic. **Leverage** — public interface (`resolve()`) stays the same, implementation consolidated. Tool config tests reuse the same helpers.

## 3. Move `claude-settings.ts` from `config/` to `features/` as adapter

**Status:** Open

- **Files**: `config/claude-settings.ts` (238 lines)
- **Problem**: `config/` is for user configuration. `claude-settings.ts` is a format adapter (Claude → OpenCode) with custom merge, fs reads, JSON parsing, event mapping. Not configuration.
- **Solution**: Move to `features/adapters/claude-settings.ts`. Config interface (`config/settings.ts`) imports the adapter, not the other way around.
- **Benefits**: **Locality** — external format adapters live together. `config/` stays about user configuration. **Leverage** — adapter has clear interface (`loadClaudeSettings(path) => Record<string, ScriptEntry[]>`).

## 4. Eliminate `features/ → audit/` dependency (cross-feature coupling)

**Status:** Open

- **Files**: `features/events/resolvers/event-config-builder.ts` (imports `audit/plugin-integration`), `features/core/toast-director.ts` (imports `audit/plugin-integration`)
- **Problem**: Two feature modules (events, core) pull global audit state. Config builder logs errors via audit; toast director logs "toast dropped" via audit. Creates invisible coupling — swapping audit requires changing events and core.
- **Solution**: Inject optional callbacks (`onError?`, `onToastDropped?`) in constructors. Entry point does the wiring.
- **Benefits**: **Leverage** — pure interfaces, no global module dependency. **Testability** — tests don't need to mock audit. **Real seam** — could swap audit system without touching events/core.

## 5. Consolidate `features/message-formatter/` from 6 files to 1-2
